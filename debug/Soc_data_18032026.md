## Bug 1 — DCache: Write bị drop khi `flush_busy` assert trong LOOKUP state

### File
`dcache_controller.v`

### Mức độ
🔴 **Critical** — gây stack corruption, CPU crash, và vòng lặp vô tận

### Mô tả
Toàn bộ sequential output block của main FSM được bao bởi guard `if (!flush_busy)`. Khi `flush_busy` assert đúng vào cycle mà `LOOKUP` state đang xử lý một write hit (`tag_lookup_stable=1`, `tag_hit=1`, `cur_we=1`), toàn bộ output — bao gồm `data_write_enable`, `tag_dirty_set`, và `cpu_ready` — bị skip hoàn toàn.

Đồng thời, combinational `next_state` logic của LOOKUP không kiểm tra `flush_busy`:

```verilog
// dcache_controller.v — next_state combinational block
DCACHE_STATE_LOOKUP: begin
    if (tag_lookup_stable) begin
        if (tag_hit)
            next_state = DCACHE_STATE_IDLE;  // chuyển IDLE bất kể flush_busy
        ...
    end
end
```

FSM chuyển về `IDLE` nhưng write chưa được thực hiện, và CPU không nhận được `cpu_ready=1` — request bị **mất hoàn toàn mà không có retry**.

### Hệ quả quan sát trong log
- `sw ra, 28(sp)` của `step5_kick_and_poll` không cập nhật data array
- `tag_dirty_set` không fire → cacheline index=45 (địa chỉ `0x10001edc`) ở trạng thái clean
- Flush FSM scan thấy `dirty=0` → bỏ qua eviction → DMEM nhận toàn số 0
- `lw ra, 28(sp)` ở epilogue: cache hit → trả về `0x00000000` → `ra=0` → `jr ra` → CPU nhảy về PC=0

### Root cause
`flush_busy` có thể assert trong khoảng 2-cycle của `LOOKUP` state khi `fence_flush` xuất hiện đúng thời điểm. Window này đủ để race condition xảy ra.

### Fix

**Bước 1** — Sửa `next_state` để LOOKUP ở lại khi write hit bị block bởi flush:

```verilog
// TRƯỚC (lỗi):
DCACHE_STATE_LOOKUP: begin
    if (tag_lookup_stable) begin
        if (tag_hit)
            next_state = DCACHE_STATE_IDLE;

// SAU (fix):
DCACHE_STATE_LOOKUP: begin
    if (tag_lookup_stable) begin
        if (tag_hit) begin
            if (!cur_we || !flush_busy)
                next_state = DCACHE_STATE_IDLE;
            // cur_we && flush_busy → ở lại LOOKUP, đợi flush_busy=0
        end
```

**Bước 2** — Tương tự cho REFILL state: `tag_update_valid` và `data_write_enable` khi `pending_write && refill_done` cũng phải không bị gate bởi `flush_busy`, hoặc dùng cơ chế tương tự để retry.

## Bug 2 — CPU Pipeline: `fence_stall` không chờ DCache flush hoàn thành

### File
`hazard_detection.v`

### Mức độ
🔴 **Critical** — gây infinite write-miss loop, simulation timeout

### Mô tả

```verilog
// hazard_detection.v dòng 43 — HIỆN TẠI (lỗi):
assign fence_stall = fence_id && !lsu_idle;
```

`fence_stall` chỉ chờ LSU idle. Khi `lsu_idle=1`:
- `fence_stall=0` → `fence_active=1` → `flush_busy=1` (flush bắt đầu)
- Pipeline **không stall thêm** — FENCE rời khỏi ID stage ngay cycle tiếp theo
- Instruction tiếp theo (`sw ra, 28(sp)`) vào LSU và bắt đầu REFILL trong DCache

Trong khi REFILL đang chạy, `flush_busy` vẫn đang ở mức `1` (flush chưa xong). Khi `refill_done` fires, toàn bộ `if (!flush_busy)` block trong REFILL state bị skip:
- `tag_update_valid = 0` → `valid[index]` không được set
- `data_write_enable = 0` → `pending_write` không commit
- `cpu_ready = 0` → CPU không nhận ack, retry vô tận

### Hệ quả quan sát trong log

```
CPU dcache req total : 19926
CPU dcache WRITE     : 19926   ← chỉ write, không có read nào hoàn thành
M1 AR handshakes     : 2491    ← 2491 lần refill, mỗi lần ~8 cycles
M1 AW handshakes     : 0       ← không có dirty line nào → write không bao giờ commit
```

CPU bị kẹt tại `sw ra, 28(sp)` — lệnh đầu tiên của `main()` ghi ra DMEM — trong suốt 500µs simulation timeout.

### Fix

**Bước 1** — Thêm input `dcache_flush_busy` vào `hazard_detection.v`:

```verilog
// Thêm input port:
input wire dcache_flush_busy,

// Sửa fence_stall:
assign fence_stall = fence_id && (!lsu_idle || dcache_flush_busy);
```

**Bước 2** — Expose `flush_busy` từ `dcache_controller.v`:

```verilog
// dcache_controller.v — thêm output port:
output wire flush_busy_out,
assign flush_busy_out = flush_busy;
```

**Bước 3** — Wire qua các tầng hierarchy:

```
dcache_controller.flush_busy_out
  → dcache_top.dcache_flush_busy  (output port)
    → soc_top (internal wire)
      → riscv_cpu_core.dcache_flush_busy  (input port)
        → hazard_detection.dcache_flush_busy  (input port)
```

## Bug 3 — DCache: AXI Write burst ghi sai data trong lần flush đầu tiên (addr `0x10000040`)

### File
`dcache_controller.v` — flush FSM output block (FLUSH_SCAN state)

### Mức độ
🟡 **High** — gây data corruption cho DMA output

### Mô tả
Trong lần flush đầu tiên (tại `t≈2845000`), AXI write burst đến địa chỉ `0x10000040` (cacheline chứa `NONCE_0` = `0xDEADBEEF`) ghi beat đầu là `0x00000000` thay vì `0xDEADBEEF`.

Lần flush thứ hai (tại `t≈5305000`) cùng địa chỉ này ghi đúng `0xDEADBEEF`.

Nguyên nhân: cacheline tại index tương ứng với `0x10000040` bị flush **trước khi** CPU kịp ghi `NONCE_0=0xDEADBEEF` vào đó. Cụ thể, `sw` của `DMEM->NONCE_0 = NONCE_0_VAL` là write miss → DCache tiến hành REFILL → trong quá trình REFILL, `fence w,w` từ `step3_write_ptext_to_dmem` fire → bug #1 xảy ra → write bị drop → cacheline có data là 0 → flush ghi 0 ra DMEM.

### Hệ quả quan sát trong log

```
[2845000] DMEM Write burst: addr=0x10000040 len=4
  beat 0: data=0x00000000  ← sai, đúng phải là 0xDEADBEEF (NONCE_0)
  beat 1: data=0xcafebabe  ← đúng (NONCE_1)
  beat 2: data=0x01234567  ← đúng (NONCE_2)
  beat 3: data=0x00000000  ← đúng (NONCE_3 chưa được ghi)
```

### Fix
Bug này là **hệ quả trực tiếp của Bug 1**. Khi Bug 1 được fix, write-allocate merge sẽ hoạt động đúng và NONCE_0 sẽ được commit vào cache trước khi flush.

## Kết quả kỳ vọng sau fix

- `M1 AW handshakes > 0` — dirty lines được evict đúng cách
- CPU tiến qua `main()` prologue và thực thi toàn bộ firmware
- Không còn RRESP=11 (SLVERR) tại `0xffffffec`
- `DMEM->NONCE_0` nhận đúng giá trị `0xDEADBEEF` sau flush
