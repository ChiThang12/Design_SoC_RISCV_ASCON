# BUG REPORT — RISC-V SoC + ASCON Crypto IP
## Dự án: Design_SoC_RISCV_ASCON
## Ngày: 2026-03-19 | Trạng thái: CPU pipeline vẫn còn bug, ASCON chưa chạy được

---

## 1. KIẾN TRÚC TỔNG QUAN

```
SoC: 3-Master × 5-Slave AXI4 crossbar
  M0 = ICache
  M1 = DCache
  M2 = DMA (width_conv 64→32)

Address map:
  S0 IMEM:    0x00000000–0x00001FFF  (8KB)
  S1 DMEM:    0x10000000–0x10001FFF  (8KB)
  S2 ASCON:   0x20000000–0x20000FFF  (4KB, MMIO NC)
  S3 SoCCtrl: 0x30000000–0x30000FFF
  S4 CLINT:   0x40000000–0x4000FFFF

CPU: 5-stage RISC-V pipeline (IF/ID/EX/MEM/WB)
  - Out-of-order stores via LSU Store Buffer (SB_DEPTH=8)
  - In-order loads via Load Queue (LQ_DEPTH=4)
  - Load scoreboard để track in-flight loads
  - DCache: Write-Back + Write-Allocate, direct-mapped, 64 sets, 16B lines
  - addr[31:29]=000 → cacheable; khác → NC (MMIO bypass)
```

## 3. BUG CHƯA FIX — `riscv_cpu_core_v2.v`

### BUG-CPU-1 ★★★ BLOCKING — EX/MEM stall gate SAI

**File:** `riscv_cpu_core_v2.v`
**Vấn đề:** EX/MEM pipeline register freeze sai điều kiện.

```verilog
// TRẠNG THÁI HIỆN TẠI (đã được fix một phần):
wire stall_ex_mem = lsu_dependency_stall_w | fence_stall;

always @(posedge clk or posedge rst) begin
    if (rst) begin ... end
    else if (!stall_ex_mem) begin   // ← ĐÃ FIX từ !stall_if → !stall_ex_mem
        regwrite_ex_mem   <= regwrite_ex;
        ...
    end
end
```

**Wire được thêm vào:**
```verilog
wire lsu_dependency_stall_w = (rs1_id != 5'b0 && lsu_scoreboard[rs1_id]) ||
                               (rs2_id != 5'b0 && lsu_scoreboard[rs2_id]);
wire stall_ex_mem = lsu_dependency_stall_w | fence_stall;
```

**Lý do:** EX/MEM chỉ freeze khi có in-flight load dependency hoặc fence drain.
Load-use hazard (load ở EX) phải cho EX/MEM advance để load vào MEM, còn ID/EX được flush (NOP).

### BUG-CPU-2 ★★★ BLOCKING — MEM/WB register overwrite sau lsu_result commit

**File:** `riscv_cpu_core_v2.v`
**Đây là bug chính hiện tại khiến ASCON không chạy được.**

**Mô tả:**
Khi `lsu_result_valid=1` (load hoàn thành) commit vào WB trong khi `stall_any=1`,
cycle tiếp theo khi `stall_any=0`, nhánh `else-if` fires và **overwrite** load result
bằng ALU result của instruction mới ở MEM stage.

```verilog
// TRẠNG THÁI HIỆN TẠI (có lsu_committed_r guard):
reg lsu_committed_r;
always @(posedge clk or posedge rst) begin
    if (rst) lsu_committed_r <= 1'b0;
    else     lsu_committed_r <= lsu_result_valid && !stall_if;
end

always @(posedge clk or posedge rst) begin
    if (rst) begin ... end
    else begin
        if (lsu_result_valid && !stall_if) begin       // nhánh 1: commit load
            mem_data_mem_wb   <= lsu_result_data;
            rd_mem_wb         <= lsu_result_rd;
            regwrite_mem_wb   <= 1'b1;
            memtoreg_mem_wb   <= 1'b1;
            jump_mem_wb       <= 1'b0;
            alu_result_mem_wb <= alu_result_mem;
            pc_plus_4_mem_wb  <= pc_plus_4_mem;
        end else if (!stall_any && !lsu_committed_r) begin  // nhánh 2: commit ALU
            alu_result_mem_wb <= alu_result_mem;
            pc_plus_4_mem_wb  <= pc_plus_4_mem;
            regwrite_mem_wb   <= regwrite_mem & ~memread_mem;
            memtoreg_mem_wb   <= 1'b0;
            jump_mem_wb       <= jump_mem;
            rd_mem_wb         <= rd_mem;
        end
    end
end
```

**Vấn đề còn lại:** Dù đã có `lsu_committed_r` guard, TC08/TC12 vẫn fail trong unit test CPU.
Pattern fail: `lw x2, 0(x1)` → `addi x3, x2, 1` → x2=0 (load không được ghi vào register file).

**Phân tích sâu hơn:**
- `lsu_result_valid && !stall_if` gate vẫn đúng về nguyên tắc (tránh deadlock)
- Nhưng `lsu_committed_r` chặn `else-if` chỉ 1 cycle — có thể chưa đủ trong mọi trường hợp
- Cần phân tích thêm: khi nào `else-if` cần bị chặn nhiều hơn 1 cycle

**Test case đang fail:**
```
TC08: lui x1,0x10000 / lw x2,0(x1) / addi x3,x2,1 / addi x4,x2,2
      Expected: x2=0xCAFEBABE, x3=0xCAFEBABF
      Got:      x2=0x00000000, x3=0x00000001

TC12: tương tự với dcache miss
TC13: SW→LW chain qua 4 NOPs — x7 (load result) = 0
```

### BUG-CPU-3 ★★ — lsu_req_valid double-enqueue (ĐÃ FIX nhưng cần verify)

**File:** `riscv_cpu_core_v2.v`
**Trạng thái hiện tại:**
```verilog
reg lsu_req_sent;
always @(posedge clk or posedge rst) begin
    if (rst) lsu_req_sent <= 1'b0;
    else begin
        if (!stall_any)                        lsu_req_sent <= 1'b0;  // CLEAR
        else if (lsu_req_valid && lsu_req_ready) lsu_req_sent <= 1'b1; // SET
    end
end
assign lsu_req_valid = (memread_mem | memwrite_mem) & !(lsu_req_sent & stall_any);
```

Fix dùng `!(lsu_req_sent & stall_any)` thay vì `!lsu_req_sent` để:
- Khi `stall_any=0`: luôn valid (pipeline advance = instruction mới)
- Khi `stall_any=1`: chỉ valid nếu chưa gửi (ngăn double-enqueue)

## 4. BUGS ĐÃ FIX HOÀN TOÀN

### BUG-DCACHE-1 ✅ — dcache_controller.v FLUSH_TAG_WAIT

**File:** `dcache_controller.v`
**Fix:** Thêm state `FLUSH_TAG_WAIT` (3'd5) giữa `FLUSH_SETTLE` và `FLUSH_EVICT`.
Tag array dùng synchronous read (1-cycle latency). State cũ đọc `tag_evict_tag_out`
1 cycle sớm → `evict_addr` sai → `AWADDR=0xFFFFFFEC` → DECERR cascade.

```verilog
`define FLUSH_TAG_WAIT 3'd5

// FSM: FLUSH_SCAN → FLUSH_SETTLE → FLUSH_TAG_WAIT → FLUSH_EVICT → FLUSH_DONE
`FLUSH_SETTLE: begin
    evict_data_0..3 <= data_read_word_0..3;  // data valid now
    flush_state <= `FLUSH_TAG_WAIT;
end
`FLUSH_TAG_WAIT: begin
    evict_addr <= {tag_evict_tag_out, flush_index, 4'b0000};  // tag valid now
    flush_state <= `FLUSH_EVICT;
end
```

### BUG-ASCON-1 ✅ — ascon_axi_slave.v core_start trong DMA mode

**Fix:** `core_start` bị suppressed khi `DMA_EN=1`:
```verilog
// Trước: if (!core_busy) core_start <= 1'b1;
// Sau:   if (!core_busy && !wr_exec_data[2] && !reg_dma_en) core_start <= 1'b1;
```

### BUG-FIRMWARE-1 ✅ — main.c DMA_LEN=24 → 8

`ASCON->DMA_LEN = DMA_PTEXT_LEN` (=8) thay vì `DMEM_DMA_OUTPUT_LEN` (=24).

### BUG-FIRMWARE-2 ✅ — main.c thiếu DATA_LEN write

Thêm `ASCON->DATA_LEN = DMA_PTEXT_LEN` trước khi ghi DMA registers.

### BUG-HAZARD ✅ — hazard_detection.v flush_id_ex thiếu mem_load_stall

```verilog
// Trước: assign flush_id_ex = load_use_hazard || branch_taken || fence_stall;
// Sau:   assign flush_id_ex = load_use_hazard || mem_load_stall || branch_taken || fence_stall;
```

## 6. TRẠNG THÁI SoC SIMULATION (`run_soc_ascon.v`)

**Triệu chứng cuối cùng quan sát được (log cycle 5188):**
- DECERR = 0 ✅
- KEY/NONCE ghi đúng vào ASCON ✅
- CTRL_DMA_EN ghi đúng (0x04) ✅
- DATA_LEN, DMA_SRC, DMA_DST ghi đúng ✅
- CTRL_DMA_START bị ghi sai (0x10000010 thay vì 0x05) ❌
  - Nguyên nhân: CPU pipeline bug khiến `addi x14, x0, 5` bị skip
  - x14 giữ stale value 0x10000010 (= DMA_DST_ADDR từ instruction trước)
- firmware sau đó loop lại từ đầu (main bị gọi đệ quy) ❌
  - Nguyên nhân: MEM/WB overwrite khiến epilogue step5 đọc ra corrupt từ stack
  - jalr về địa chỉ sai → gọi lại step3/main

**Root cause chain:**
```
MEM/WB overwrite bug
  → lw x1, 28(sp) đọc sai (ra bị corrupt trên stack)
  → jalr x0, 0(x1) nhảy về sai
  → firmware loop đệ quy
  → CTRL_DMA_START không bao giờ được ghi đúng
  → ASCON không start
```

## 7. HƯỚNG FIX TIẾP THEO

### Ưu tiên 1: Fix MEM/WB commit logic

Vấn đề cốt lõi: cần commit load result **kể cả khi** `stall_any=1`, nhưng phải ngăn `else-if` overwrite ngay cycle sau.

**Approach 1 (đã thử, chưa đủ):** `lsu_committed_r` 1-cycle guard
**Approach 2 (chưa thử):** Dùng `regwrite_mem_wb_lock` flag rõ hơn:

```verilog
// Khi lsu_result commit: set lock
reg lsu_wb_lock;
always @(posedge clk or posedge rst) begin
    if (rst) lsu_wb_lock <= 0;
    else if (lsu_result_valid && !stall_if && regwrite_mem_wb)
        lsu_wb_lock <= 1;
    else if (!stall_any && !lsu_result_valid)
        lsu_wb_lock <= 0;
end

// else-if chỉ fire khi không locked và không commit load
else if (!stall_any && !lsu_wb_lock) begin
    ...commit ALU result...
    lsu_wb_lock <= 0;
end
```

**Approach 3 (khuyến nghị):** Refactor MEM/WB thành state machine rõ ràng hơn:
- State IDLE: chờ
- State LOAD_PENDING: load dispatched, chờ result
- State LOAD_DONE: hold load result 1 cycle cho forwarding
- State ALU: commit ALU result

### Ưu tiên 2: Verify stall_ex_mem hoàn chỉnh

Sau khi TC08/TC12/TC13 pass, cần verify lại SoC simulation để confirm CTRL_DMA_START được ghi đúng.

## 9. TÓM TẮT NHANH CHO AGENT MỚI

**Chỉ còn 1 bug cần fix:**

File `riscv_cpu_core_v2.v`, hàm MEM/WB register.

Vấn đề: khi load hoàn thành (`lsu_result_valid=1`) và commit vào WB,
cycle tiếp theo `else-if (!stall_any)` bị fire và overwrite kết quả load.

Hiện tại đã có `lsu_committed_r` guard nhưng chưa đủ (TC08/TC12/TC13 vẫn fail).

Cần đảm bảo: sau khi `lsu_result_valid` commit, WB stage giữ nguyên
load result (`rd_mem_wb`, `regwrite_mem_wb=1`, `memtoreg_mem_wb=1`, `mem_data_mem_wb`)
cho đến khi register file đã ghi xong và forwarding đã dùng xong.
